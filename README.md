# Game Analytics

Небольшой учебный проект по продуктовой аналитике в игровой индустрии. Репозиторий объединяет две части:

- SQL-задачи по событиям пользователей и платежам.
- A/B-анализ изменения стартового предложения в игровом магазине.

Проект специально оставлен компактным: без лишней инфраструктуры, но в более аккуратном и понятном виде.

## Структура проекта

- `data/task1_facts.sql` - исходные данные для SQL-задач.
- `data/task2_ab_dataset.csv` - датасет для A/B-анализа.
- `sql/task1_queries.sql` - SQL-решения по аналитическим задачам.
- `notebooks/ab_test_analysis.ipynb` - ноутбук с исследованием и визуализациями.
- `scripts/ab_test_analysis.py` - самостоятельный Python-скрипт для расчета ключевых метрик A/B-теста.

## Что здесь разобрано

### SQL

В SQL-части решаются три задачи на MySQL:

1. Расчет выручки со 2-го по 7-й платеж пользователя.
2. Среднее число логинов и платежей на пользователя за первые 28 дней жизни.
3. Оценка среднего LTV на горизонтах 7 и 30 дней для разных дней lifetime.

### A/B-тест

Сравниваются две группы пользователей:

- группа 1 - базовое предложение;
- группа 2 - сниженная цена в 5 раз и увеличенное наполнение набора в 2 раза.

В анализе считаются и сравниваются:

- `CR` - конверсия в платящего пользователя;
- `ARPU` - средняя выручка на пользователя;
- `ARPPU` - средняя выручка на платящего пользователя.

Для проверки различий используются:

- `chi-square` для конверсии;
- `Mann-Whitney U` для денежных метрик.

## Как запустить

Установить зависимости:

```bash
pip install -r requirements.txt
```

Запустить Python-скрипт с A/B-анализом:

```bash
python scripts/ab_test_analysis.py
```

Если нужно сохранить график:

```bash
python scripts/ab_test_analysis.py --output reports/ab_test_metrics.png
```

SQL-запросы находятся в файле `sql/task1_queries.sql` и могут выполняться отдельно в MySQL.

## Краткий вывод по A/B-тесту

- Конверсия в тестовой группе растет.
- `ARPPU` снижается из-за уменьшения цены оффера.
- Итог по `ARPU` требует осторожной интерпретации: рост конверсии не обязательно перекрывает падение среднего чека.

Практический вывод: изменение не стоит выкатывать без дополнительной проверки долгосрочных эффектов, в первую очередь retention и накопленной выручки на более длинном горизонте.
